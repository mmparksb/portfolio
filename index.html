전표 출력 백그라운드 응용프로그램

회원으로부터 주문이 들어오면 거래기록, 배송정보가 담긴 영수증을 고객사에서 사용하는 SRP-350III 모델 영수증 출력기에서 즉각적으로 출력할 수 있어야하는 상황이었습니다.
처음 생각했던 방안은 ⑴해당 기기와 연결된 PC에 서버를 하나 구축하여 항상 요청을 받을 수 있도록 포트포워딩을 하는 것이었습니다.
하지만 보안에 대한 리스크가 크고, 혹시라도 서버가 요청을 받지 못했을 때, 다시 요청을 해야만 하는 문제가 있다고 생각했습니다.
다음으로 생각한 방안은 ⑵DB에 출력 요청을 저장하고, 이를 js로 하나씩 웹페이지로 요청을 받아와서 전표 출력 명령을 전달하는 방법이었습니다.
이 방법은 전표 출력기와 연결된 PC가 여러 사람이 사용하고, 컴퓨터에 익숙하지 않은 분들이어서 웹 브라우저를 닫으면 동작하지 않는 문제가 있습니다.
결국 최종적으로 생각해낸 방법은 ⑶백그라운드 응용프로그램을 만드는 것이었습니다.
Python의 PyInstaller 라이브러리를 사용하여 크롤링 실행을 누구나 할 수 있게 exe 실행 파일로 만들어서 공유한 경험이 있어서 백그라운드 응용프로그램도 만들 수 있겠다 생각했습니다.
그리하여 큐 구조를 생각하며 요청은 모두 한 테이블에 저장하고, 이를 주기적으로 조회하여 명령을 전달하는 프로그램을 만들었습니다.
프로그램을 만들어 고객사에 방문하였고, Windows 10 의존성 문제로 vcruntime140.dll 파일이 없다며 Windows 7 운영체제였던 컴퓨터에서 실행이 되지 않았습니다.
해당 컴퓨터는 24시간 계속해서 가동되어야 하는 상황이라 중간에 재부팅을 할 수 없는 상황이었고,
그 자리에서 Python 버전을 3.12.7에서 3.7.16으로 다운그레이드하여 재빌드 하여 문제를 해결하였습니다.
해당 전표 출력 프로그램은 지금도 문제 한 번 없이 잘 돌아가고 있어서 굉장히 뿌듯한 경험입니다.

온라인 쇼핑몰 - 오프라인 매장 포스기 데이터 연동

새로 CAFE24로 구축한 온라인 쇼핑몰에 등록한 상품, 발생한 주문 데이터를 기존에 운영중이던 오프라인 매장에서 사용하고 있는 POS기와 연동을 해야하는 상황이었습니다.
POS기 개발사에서는 온라인 쇼핑몰처럼 다른 곳에서 발생하는 데이터를 쉽게 사용할 수 있는 API가 준비되지 않은 상태였고,
그렇게 담당 개발자와 함께 외부 데이터를 위한 API를 하나씩 만들어가기 시작했습니다.
기존에는 온라인 쇼핑몰에도 상품을 등록하고, POS기에도 상품을 등록하셔서 두 번 등록해야 했지만,
CAFE24 API를 사용하여 상품 데이터를 보내주면, POS기에도 즉각적으로 상품을 연동할 수 있게 하였습니다.
또한 CAFE24 개발자센터에서 웹 훅을 사용하여 주문이 발생하면 json 형식으로 넘어온 주문 데이터를 받고, API를 사용하여 추가적인 주문상세 정보를 조회하였고,
이를 POS기 API를 사용하여 바로 주문 데이터가 들어가도록 개발을 진행하였습니다.
이 때의 협업을 통해, 매장별 재고를 지원하지 않는 CAFE24 쇼핑몰에서 ajax를 사용하여 실시간으로 매장별 재고를 볼 수 있는 기능도 추가 개발할 수 있었습니다.
확장성 있는 코드로 당시 1개뿐이었던 매장이 2호점이 생긴 이후에도 추가 개발이나 보완 없이 잘 동작하였습니다.

100만 단위 규모의 데이터 이전

회원 5000, 단체고객 10000, 주문 600000, 결제 700000, 장소 10000, 주문 상세 내역 600000, 배송 정보 800000 가 포함된
약 50개의 테이블이 있는 DB였습니다. 기존 asp + MS SQL에서 그누보드(PHP) + MySQL로 이전할 때, 그누보드의 회원, 상품, 게시판 테이블을 사용해야 하는 상황이었습니다.
첫 데이터가 10년이 넘었고, 어떤 테이블은 칼럼이 60개가 되었으며, 데이터가 일정하지 않은 것을 보아하니 중간에 변동된 점이 많았습니다.
이를 분석하고 Python의 pandas 라이브러를 사용하여 결측치를 파악하고, 잘못된 데이터와 함께 재가공하였습니다.
참조관계를 설정하고 다양한 쿼리문을 테스트 해보았을 때, 속도가 느려 몇몇 INDEX 설정까지 마친 경험이 있습니다.
해당 경험을 통하여 카페24 데이터 이전 팀과 협력을 하게 되었고, 더 많은 고객사와 접할 수 있게되었습니다.

데이터 크롤링, 데이터 입력

와인 정보를 공유하는 사이트에서 평점과 평가자 수, 어울리는 음식 등의 데이터를 크롤링하여 온라인 쇼핑몰(CAFE24 기반)에 올라와 있는 상품에 데이터를 입력해야 하는 상황이었습니다.
처음에는 전체 상품을 크롤링하여 DB에 모든 데이터를 저장하고 이를 CAFE24 API를 활용하여 필요한 데이터를 상품 데이터에 넣어주었습니다.
이는 두 가지 문제점이 있었는데, 하나는 전체 상품 목록에서의 페이지네이션이 80 페이지가 넘어가는 순간부터는 항상 동일한 상품을 보여주고 있었습니다.
DB와 비교를 해보았더니 검색을 했을 때 나오는 상품이 크롤링 하는 과정에서는 보이지 않았습니다. 이로 인해 원하는 상품의 데이터를 가져올 수 없었습니다.
다른 문제점은 CAFE24 API는 초당 2회의 콜수 제한이 있어 천 단위의 상품 데이터를 입력하는 과정도 시간 소요가 컸습니다.
이 두 문제를 해결하기 위해 크롤링 프로그램을 다시 만들었습니다.
웹훅을 사용하여 상품이 등록, 수정될 때마다 DB에 상품명을 저장해두고
크롤링을 할 때 전체 데이터를 가져오는 것이 아닌, DB에 존재하는 상품을 직접 검색하여 데이터를 수집하는 형식으로 개선하였습니다.
또한 데이터를 수집하고 바로 CAFE24 API로 상품 데이터를 입력해주었습니다.
이를 통해 기존 20건밖에 매칭되지 못했던 데이터가 100건 넘게 매칭되도록 개선되었으며
전체 데이터가 아닌 등록된 데이터만 탐색하고, API 콜수 제한을 대비하여 즉각적으로 데이터를 넣기 때문에 전체적인 시간도 감소되었습니다.
또한 Python의 PyInstaller로 exe 응용프로그램으로 만들어 누구나 크롤링과 데이터 업데이트가 가능하였습니다.

1500~2500명 규모의 Fax, Sms, Email 발송

온라인 쇼핑몰 운영하시는 분께서 원하는 타이밍에 잔금이 처리되지 않은 고객을 대상으로 전자세금거래서를 단체 발송해야하는 상황이었습니다.
POPBiLL의 API를 통해 팩스와 SMS를 보낼 수 있었고, 버튼을 누르면 발송되는 것으로 개발하기로 하였습니다.
버튼을 누르고 결과를 볼 수 있게끔 개발하기 위하여, 500명씩 끊어서 각 세금계산서 excel 파일을 만들고, POPBiLL API로 전승을 요청하는 API를 새로 개발하였습니다.
하지만 요청의 처리가 3분을 넘어가면 해당 호스팅 서버에서 404에러를 띄우며 중단시켰습니다.
점차 수를 줄여가며 2분정도로 각각 400건, 300건으로 설정하였습니다.
마찬가지로 SMTP를 설정하고 이메일 발송도 20건으로 설정하여 문제를 해결하였습니다.
